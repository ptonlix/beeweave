# Setup

A skill-based framework for AI coding agents — Claude Code, Cursor, Windsurf, Pi, Gemini CLI, Google Antigravity, Codex, Hermes, OpenClaw, OpenCode, Aider, Factory Droid, Trae / Trae CN, Kiro, GitHub Copilot (CLI + VS Code Chat) — to build and maintain an Obsidian wiki using Karpathy's LLM Wiki pattern. No scripts, no API keys — the agent **is** the LLM.

> Running `bash setup.sh` lets you choose optional extra global skills and agents, then wires selected project-local skill directories with the full skill set (`.claude/skills/`, `.codex/skills/`, `.cursor/skills/`, `.windsurf/skills/`, `.agents/skills/`, `.pi/skills/`, `.kiro/skills/`) and selected global skill directories with portable skills only. Global installs default to `beeweave-update`, `beeweave-query`, and `beeweave-ingest`; advanced extras can be added explicitly with `--global-extra beeweave-capture,beeweave-context-pack`. Use `bash setup.sh --agents claude,codex` for a compact non-interactive selection, or `bash setup.sh --agents all` for every supported agent. See the [Agent Compatibility table in README.md](README.md#agent-compatibility) for the full matrix.

## Install via pip (no clone needed)

```bash
pip install beeweave
bwe setup --vault /path/to/your/vault
```

The packaged CLI writes `~/.beeweave/config`, creates the standard `vault/`
and simplified `workbench/` skeleton at the locations you choose, and lets
you choose which agent skill directories to install. In an interactive terminal
it shows a numbered picker; for non-interactive installs use `--agents
claude,codex`, `--agents all`, or `--agents none`. Global installs are
portable-only by default (`beeweave-update`, `beeweave-query`, `beeweave-ingest`), while project-local installs get
the full skill set. Project-local setup also generates `AGENTS.md`, `CLAUDE.md`,
`GEMINI.md`, and `HERMES.md` in the target project, with the agent-specific
files pointing to `AGENTS.md`. Use `--copy` to copy skill files instead of symlinking. Run
`bwe info` to see resolved paths.

To uninstall the packaged CLI wiring, run `bwe uninstall --yes`. This removes
BeeWeave-managed skills from global agent directories, removes project-local
BeeWeave skills/bootstrap files in the current project, and deletes
`~/.beeweave` config files. Your vault and workbench content are left in place.
Use `--project /path/to/project` to clean a specific project, or
`--no-project-local` to only remove global installs and config.

The rest of this doc covers the `git clone` + `setup.sh` path.

When you are working from a source checkout and have not installed the package,
use `python3 -m beeweave ...` for the helper CLI. Once installed with
`pip install -e .` or `pip install beeweave`, the `bwe ...`
entrypoint is available on PATH.

## Quick Start (git clone)

### 1. Set your vault path

```bash
cp .env.example .env
```

Open `.env` and set `BEEWEAVE_VAULT_PATH` to your Obsidian vault. The default points at `./vault`, which setup creates as a runtime directory in your chosen project, not as a committed repository fixture:

```
BEEWEAVE_VAULT_PATH=./vault
BEEWEAVE_WORKBENCH_PATH=./workbench
```

That's the only required config.

## Monorepo Workbench

The git-clone layout contains BeeWeave source files only. Runtime `vault/` and
`workbench/` directories are generated by setup in the project/vault locations
you choose.

The generated knowledge workspace layout is:

```
<project>/
└── workbench/     # inbox, article workspace, source library

<vault>/
└── ...            # compiled Obsidian wiki knowledge
```

Use `workbench/` for writing and exploration:

- `workbench/inbox/captures/` — Agent quick captures and stop-hook findings
- `workbench/inbox/web/` — browser captures and web clippings
- `workbench/inbox/archived/` — processed original inputs
- `workbench/inbox/rejected/` — rejected staged pages and patches
- `workbench/articles/drafts/` — active article drafts
- `workbench/articles/published/` — finished article copies or publication records
- `workbench/library/` — source notes, copied references, reading material

Use `vault/` for internalized knowledge maintained by the wiki skills. The
intended loop is: query the vault, write in the workbench, ingest stable knowledge
back into the vault, then query again for the next draft.

`.skills/workbench/` is the source of truth for creation skills, while
`.skills/wiki/` contains wiki maintenance skills. `setup.sh` symlinks both groups
into selected project-local agent directories, while global installs remain
portable-only by default (`beeweave-update`, `beeweave-query`, `beeweave-ingest`) with optional extras via `--global-extra`.

### 2. Point an agent at the skills

Open this project in your coding agent and tell it what you want:

| What you say | Skill triggered |
|---|---|
| "Set up my wiki" | `beeweave-setup` |
| "Ingest my documents from ~/research" | `beeweave-ingest` |
| "/beeweave-history-ingest claude" or "/beeweave-history-ingest codex" | `beeweave-history-ingest` |
| "Import my Claude history" | `beeweave-claude-ingest` |
| "Import my Codex history" | `beeweave-codex-ingest` |
| "Import my Pi history" | `beeweave-pi-ingest` |
| "Process this ChatGPT export" | `beeweave-ingest` |
| "What's the status of my wiki?" | `beeweave-status` |
| "What do I know about X?" | `beeweave-query` |
| "Audit my wiki" | `beeweave-lint` |
| "Rebuild from scratch" | `beeweave-rebuild` |

The agent reads skills from `.skills/wiki/` and `.skills/workbench/`, resolves the full BeeWeave profile via the Config Resolution Protocol (`@name` override → `.env` → `~/.beeweave/config`), and does the work.

### 3. Open in Obsidian

Open your vault directory in Obsidian (File → Open Vault). The wiki pages, wikilinks, and graph view all work natively.

## What Can It Ingest?

Anything text-based:

| Source | Skill | What it reads |
|---|---|---|
| Markdown, PDFs, text files | `beeweave-ingest` | Any document directory |
| Claude Code history | `beeweave-claude-ingest` | `~/.claude/` — conversations, memories, sessions |
| Codex CLI history | `beeweave-codex-ingest` | `~/.codex/` — sessions, rollouts, history index |
| Pi agent sessions | `beeweave-pi-ingest` | `~/.pi/agent/sessions/` — tree-structured JSONL |
| ChatGPT exports | `beeweave-ingest` | `conversations.json` from ChatGPT export |
| Slack / Discord logs | `beeweave-ingest` | Channel export JSON files |
| Meeting transcripts | `beeweave-ingest` | Any text transcript |
| Raw text dumps | `beeweave-ingest` | Anything — CSV, logs, notes |

## Tracking & Delta

The framework tracks everything it ingests via `.manifest.json` in the vault root. This enables:

- **Status view** — "What's been ingested? What's new? What's changed?"
- **Delta ingestion** — Only process new/modified sources, skip what's already in the wiki
- **Provenance** — Which source produced which wiki page
- **Staleness detection** — Source changed but wiki page hasn't been updated

### Typical workflow

```
"What's the status?"     → beeweave-status computes the delta
"Ingest the new stuff"   → beeweave-ingest processes only the delta (append mode)
"What's the status now?" → beeweave-status confirms everything is up to date
```

### When things drift too far

```
"Archive and rebuild"    → beeweave-rebuild archives current wiki to _archives/, clears, ready for fresh ingest
"Restore the old one"    → beeweave-rebuild restores from a previous archive
```

Archives live at `$VAULT/_archives/` with full snapshots. Nothing is ever lost.

## Vault Structure

```
$BEEWEAVE_VAULT_PATH/
├── concepts/           # Global knowledge — ideas, theories, mental models
├── entities/           # People, orgs, tools
├── skills/             # How-to knowledge, procedures
├── references/         # Source summaries
├── synthesis/          # Cross-cutting analysis
├── projects/           # Per-project knowledge
│   ├── my-project/
│   │   ├── my-project.md
│   │   ├── concepts/
│   │   └── skills/
│   └── another-project/
│       └── ...
├── _meta/              # Taxonomy, dashboards, Obsidian Bases
├── _archives/          # Wiki snapshots for rebuild/restore
├── _staging/           # Review queue when staged writes are enabled
├── .obsidian/          # Obsidian app configuration
├── hot.md              # Recent semantic cache
├── index.md            # Auto-maintained catalog
├── log.md              # Chronological operation log
└── .manifest.json      # Ingest tracking ledger
```

Knowledge that's project-specific goes under `projects/<name>/`. Knowledge that's general goes in the global category directories. Both are cross-referenced with `[[wikilinks]]`.

## Optional Config

| Variable | What it does | Default |
|---|---|---|
| `BEEWEAVE_WORKBENCH_PATH` | Workbench root for captures, source material, and drafts | `./workbench` |
| `BEEWEAVE_CATEGORIES` | Wiki page categories | `concepts,entities,skills,references,synthesis,projects` |
| `BEEWEAVE_MAX_PAGES_PER_INGEST` | Max pages updated per ingest | `15` |
| `CLAUDE_HISTORY_PATH` | Where to find Claude data | *auto-discovers from `~/.claude`* |
| `CODEX_HISTORY_PATH` | Where to find Codex data | *defaults to `~/.codex`* |
| `PI_HISTORY_PATH` | Where to find Pi sessions | *defaults to `~/.pi/agent/sessions`* |
| `LINT_SCHEDULE` | Wiki health check frequency | `weekly` |

| Skill | Purpose |
|---|---|
| `beeweave-core` | Core pattern — 3-layer architecture, page templates, project org |
| `beeweave-setup` | Initialize vault structure, create index/log, configure Obsidian |
| `beeweave-ingest` | Distill source documents into wiki pages (append or full mode), plus any raw text — chat exports, logs, transcripts, anything |
| `beeweave-history-ingest` | Unified history ingest router (`claude`, `codex`, `pi`) |
| `beeweave-claude-ingest` | Mine `~/.claude` conversations and memories into wiki pages |
| `beeweave-codex-ingest` | Mine `~/.codex` sessions and rollout logs into wiki pages |
| `beeweave-pi-ingest` | Mine `~/.pi/agent/sessions` JSONL history into wiki pages |
| `beeweave-status` | Audit: what's ingested, what's pending, delta, recommend action |
| `beeweave-rebuild` | Archive current wiki, rebuild from scratch, or restore from archive |
| `beeweave-query` | Answer questions from the compiled wiki with citations |
| `beeweave-lint` | Find orphans, broken links, stale content, contradictions |
| `beeweave-update` | Sync current project's knowledge into the vault (works from any project) |
| `beeweave-skill-creator` | Create new skills to extend the framework |

## How It Works

No scripts, no dependencies. The skills are markdown files that tell an AI agent *how* to operate on your Obsidian vault:

1. Agent reads `.env` for vault path
2. Agent reads `.manifest.json` to know what's already been done
3. Agent reads the relevant skill for instructions
4. Agent uses its built-in tools (read, write, search) to do the work
5. Agent updates `.manifest.json` to track what it did
6. Output is standard Obsidian-compatible markdown with frontmatter and `[[wikilinks]]`

**The wiki is the artifact. The agent is the maintainer. Obsidian is the viewer.**

## Extending

Want a new workflow? Use the `beeweave-skill-creator` skill:

> "Create a skill that generates weekly summaries from my workbench drafts"

It walks you through drafting, testing, and refining a new skill under `.skills/wiki/` or `.skills/workbench/`.
